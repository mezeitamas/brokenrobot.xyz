# Commit conventions

Commits follow [Conventional Commits](https://www.conventionalcommits.org/). A consistent
history keeps the log readable and makes it easy to see _why_ the site changed over time.

## Format

```
type(scope): description

optional body

optional footer(s)
```

The `type` and `description` are required; `scope`, body, and footers are optional.

## Types

| Type       | Use for                                                          |
| ---------- | ---------------------------------------------------------------- |
| `feat`     | A new user-facing feature or capability.                         |
| `fix`      | A bug fix.                                                       |
| `post`     | Publishing or updating a blog article (this site's custom type). |
| `docs`     | Documentation only (e.g. the files in `docs/`, the README).      |
| `style`    | Formatting/whitespace with no behaviour change.                  |
| `refactor` | Code change that neither fixes a bug nor adds a feature.         |
| `perf`     | A change that improves performance.                              |
| `test`     | Adding or adjusting tests.                                       |
| `build`    | Build system or dependency changes.                              |
| `ci`       | CI/CD pipeline changes.                                          |
| `chore`    | Maintenance that doesn't fit the above (e.g. routine upkeep).    |

## Scope

The scope is optional and, when present, lowercase. Prefer a recognizable area of the
codebase — for example `blog`, `rss`, `layout`, `seo`, `styles`, `content`, `deps`, `ci`.

```
feat(layout): add a light/dark theme toggle
post(blog): the renaissance of written coding conventions
chore(deps): update dependencies to the latest
```

## Subject line

- Imperative mood ("add", "fix", "restore" — not "added" or "adds").
- Start lowercase, no trailing period.
- Keep it short and specific; the subject is a summary, not the whole story.

## Body — why, not what

The body answers **why**, not **what**. The diff already shows what changed. Use short prose
to explain the motivation — the problem, the constraint, or the reasoning — not a restatement
of the changes.

**Good** — short prose, explains motivation:

```
fix(rss): use absolute article URLs in the feed

Entries pointed at relative paths, so opening a post from an RSS
client landed on a broken link. Building absolute URLs from the
configured site keeps the feed usable in every reader.
```

**Bad** — bullet-list changelog restating the diff:

```
fix(rss): use absolute article URLs in the feed

- Imported SITE_METADATA in rss.xml.ts
- Mapped posts to absolute URLs
- Updated the RSS test snapshot
- Renamed a loop variable
```

If you find yourself writing more than three sentences, the commit is probably doing too
much — consider splitting it.

## Breaking changes

Mark a breaking change with a `!` after the type/scope, and/or a `BREAKING CHANGE:` footer
describing the impact.

```
feat(content)!: require a `summary` field on blog frontmatter

BREAKING CHANGE: posts without a `summary` no longer build.
```

## Footers

Footers are optional and go at the end, one per line — for example referencing an issue:

```
Refs #123
```

Do **not** add authorship or co-author trailers (e.g. `Co-Authored-By:`, `Signed-off-by:`),
including ones generated by AI coding tools. Commits record the change and its motivation,
not who or what typed them.
